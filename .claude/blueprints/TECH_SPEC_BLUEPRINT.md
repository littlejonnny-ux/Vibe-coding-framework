# Blueprint: Генерация Technical Specification проекта

---

## Когда использовать

При инициализации нового проекта или при необходимости формализовать бизнес-логику. Пользователь описывает механики, Claude Code структурирует в ТЗ по этому шаблону.

---

## Принципы ТЗ

1. **Живой документ.** Обновляется при каждом значимом изменении механик. Не отдельные migration-файлы — правки вносятся напрямую + строка в «Историю изменений».
2. **Source of truth.** Если ТЗ и код расходятся — ТЗ обновляется. Код реализует ТЗ, не наоборот.
3. **Уровень детализации.** Достаточный для реализации без додумывания. Формулы, поля БД, роли, статусы — конкретно.
4. **Независимость от стека.** ТЗ описывает ЧТО, не КАК. Технический стек — в CLAUDE.md.

## Правила обновления

1. **Когда обновлять:** При изменении бизнес-логики, добавлении/удалении механик, изменении формул, ролей, статусов, таблиц БД.

2. **Как обновлять:** Изменить соответствующий раздел напрямую. Добавить строку в «Историю изменений». Не создавать отдельных файлов-миграций.

3. **Кто обновляет:** Claude Code автоматически через /update-docs после merge, если были изменения бизнес-логики.

4. **Версионирование:** MAJOR.MINOR. MAJOR — при добавлении новой крупной механики. MINOR — при изменении существующей.

## Уровни детализации

### Для каждой формулы:
- Сама формула (математическая запись или псевдокод)
- Объяснение каждого компонента
- Источник каждого компонента (из какой таблицы/поля)
- Edge cases (деление на ноль, null-значения, пустые массивы)
- Пример расчёта с конкретными числами

### Для каждой таблицы БД:
- Все поля с типами
- FK-связи
- Обязательность (NOT NULL)
- Defaults
- Constraints

### Для каждого workflow:
- Пошаговый процесс
- Кто выполняет каждый шаг
- Что происходит при ошибке
- Автоматические действия (auto-approve, auto-calculate)

### Для каждой роли:
- Что видит (полный список экранов)
- Что может делать (CRUD-операции по сущностям)
- Что НЕ может делать (явные запреты)
