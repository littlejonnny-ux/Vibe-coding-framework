# Архитектурные принципы кода

> Принципы 1–7: frontend. Принципы 8–11: backend.
> Все принципы универсальны — применяются к любому проекту, любому стеку, любому тиру.
> Проектные конкретизации (папки, файлы, инструменты) — в CLAUDE.md проекта.

---

## Frontend

### Принцип 1 — Декомпозиция (ALL tiers)

Компонент или функция — не более 300 строк. Файл — не более 800 строк. При превышении — обязательная декомпозиция на подкомпоненты/подфункции в той же директории.

Страница = композиция подкомпонентов, не монолит. Один файл-оркестратор (загрузка данных, состояние) + набор подкомпонентов (UI-блоки).

### Принцип 2 — Единственная ответственность (ALL tiers)

Каждый компонент, файл и функция отвечают за одну вещь:
- Один компонент = один UI-блок (не страница целиком)
- Одна функция = одна операция
- Один файл = один компонент или одна группа связанных утилит

Проверка: если описание компонента требует союза «и» — он делает слишком много.

### Принцип 3 — Разделение логики и отображения (ALL tiers)

- Data fetching — в отдельных функциях, хуках или серверных слоях, не внутри JSX/шаблонов
- Расчёты и бизнес-логика — в выделенных файлах (calculations, utils), не внутри компонентов
- Константы — в едином файле констант, не дублировать между файлами

### Принцип 4 — Переиспользование (ALL tiers)

- Если UI-паттерн используется в 2+ местах — вынести в общий компонент
- Общие компоненты — в отдельной директории (shared/ui)
- Утилитарные функции — в отдельной директории (lib/utils)

### Принцип 5 — Независимость компонентов (ALL tiers)

- Подкомпонент НЕ вызывает data source напрямую — получает данные через props или хуки
- Компонент принимает минимум props — только те данные, которые нужны для отображения
- Компонент не знает о внутреннем устройстве соседних компонентов
- Компонент можно удалить или заменить, не трогая другие компоненты

### Принцип 6 — Типизированные контракты (STANDARD+)

Между слоями приложения (data source → бизнес-логика → UI) взаимодействие через явные типизированные интерфейсы, не через «сырые» объекты из data source.

- Для каждой бизнес-сущности — ViewModel-тип (что нужно UI)
- Mapper-функция преобразует raw data → ViewModel
- Компоненты принимают ViewModel, не Row из БД
- Изменение схемы БД локализуется в одном mapper-файле

```
БД добавляет поле `archived_at`
  → Row тип изменился (автогенерация)
  → Mapper toViewModel() обновлён — одно место
  → ViewModel НЕ изменился → компоненты НЕ ломаются
```

### Принцип 7 — Модульные границы (STANDARD+)

Кодовая база организуется по бизнес-доменным модулям (feature-based), не по техническим типам.

**Кто определяет границы:** Claude Code. При генерации CLAUDE.md анализирует бизнес-домен и определяет модули. Пользователь описывает домен, не структуру.

**Правила:**
- Каждый модуль — папка с компонентами, хуками, типами, тестами этой фичи
- Модуль экспортирует наружу только public API (index.ts)
- Импорт из внутренностей чужого модуля запрещён
- Shared-компоненты (2+ модулей) — в отдельном модуле `shared/`

## Backend

### Принцип 8 — Атомарность серверных операций (ALL tiers)

Связанные изменения в БД — одна транзакция или одна PostgreSQL-функция. Не цепочка отдельных запросов, каждый из которых может упасть.

Пример: утверждение карты (обновление статуса + audit_log + пересчёт total) = одна транзакция.

### Принцип 9 — Серверная валидация как source of truth (ALL tiers)

- Клиент валидирует для UX (быстрая обратная связь)
- Сервер валидирует для безопасности (единственный гарант)
- Никогда не доверять данным от клиента — даже если на клиенте Zod-схема

### Принцип 10 — Изоляция data access (ALL tiers)

Запросы к БД — в выделенных функциях или hooks, не разбросаны по handlers/components. Изменение схемы БД должно затрагивать один файл data access layer, а не 15 мест в коде.

### Принцип 11 — Enforced access control на уровне БД (ALL tiers)

RLS-политики как primary gate, application code как secondary. Даже при баге в application code — БД не отдаст чужие данные.

## Чеклист перед созданием нового файла

Claude Code проверяет:
- [ ] У компонента/функции одна ответственность (описание без «и»)
- [ ] Не обращается к data source напрямую (кроме оркестраторов)
- [ ] Принимает только необходимые props/параметры (ViewModel, не raw data)
- [ ] Можно переиспользовать в другом контексте
- [ ] Файл размещён в правильном модуле
- [ ] Если используется в 2+ модулях — в shared/
- [ ] Компонент/функция ≤ 300 строк, файл ≤ 800 строк
- [ ] Серверные операции атомарны

## E2E-тестирование пользовательских сценариев (STANDARD+)

### Принцип: механика без E2E-теста — незавершённая механика

Каждая пользовательская механика, доходящая до merge, сопровождается E2E-тестом.
Тест пишется в том же PR, не откладывается.

### Что считается пользовательской механикой (и требует E2E)

- Новая страница с интерактивными элементами
- CRUD-операция (создание, редактирование, удаление сущности)
- Смена статуса (workflow: draft → active → approved)
- Форма с валидацией и отправкой данных
- Ролевое ограничение (пользователь X не должен видеть/делать Y)
- Навигационный переход (клик → открывается детальная страница)

### Что НЕ требует E2E

- Чистая бизнес-логика без UI (calculations.ts) — покрыта unit-тестами
- Data layer (hooks/) без нового UI — покрыт unit-тестами
- Стилевые правки (CSS, цвета, отступы)
- Docs-only изменения

### Структура тестов

```
src/__tests__/e2e/
├── auth.spec.ts              # Логин, роли, редиректы, смена пароля
├── dashboard.spec.ts         # Дашборды по ролям, метрики, навигация
├── kpi-library.spec.ts       # CRUD библиотеки, фильтры, копирование
├── kpi-cards.spec.ts         # Карты: создание, факт, согласование, формула
├── participants.spec.ts      # CRUD участников, импорт, сброс пароля
├── trigger-goals.spec.ts     # CRUD запускающих целей
├── dictionaries.spec.ts      # CRUD справочников
├── negative.spec.ts          # Негативные сценарии (запреты, валидация)
└── helpers/
    ├── auth.helper.ts        # Логин под разными ролями
    ├── seed.helper.ts        # Подготовка тестовых данных
    └── constants.ts          # Тестовые credentials, URLs
```

### Формат E2E-теста

```typescript
test('Participant вводит факт binary — % исполнения обновляется', async ({ page }) => {
  // Arrange: логин как participant, открыть карту с binary KPI
  // Act: нажать "Да"
  // Assert: % исполнения = 100%, визуально зелёный progress bar
});
```

### Когда запускать — трёхуровневая схема

Claude Code определяет уровень автоматически на основе `git diff` перед merge.

#### Уровень 1 — Пропуск E2E

**Условие:** PR не затрагивает UI. Изменения только в:
- calculations.ts, constants.ts, utils.ts
- hooks/ без новых компонентов
- .md файлы, конфиги, CSS-only

**Действие:** E2E не запускаются. Счётчик merges_without_full_e2e += 1.

#### Уровень 2 — Targeted E2E

**Условие:** PR затрагивает UI конкретного модуля.

**Действие:** запустить E2E-тесты только затронутого модуля:
- Изменения в features/kpi-cards/ → `npx playwright test kpi-cards.spec.ts`
- Изменения в features/kpi-library/ → `npx playwright test kpi-library.spec.ts`
- Изменения в features/participants/ → `npx playwright test participants.spec.ts`
- Изменения в features/trigger-goals/ → `npx playwright test trigger-goals.spec.ts`
- Изменения в features/shared/ или app/(dashboard)/ → `npx playwright test dashboard.spec.ts`

Счётчик merges_without_full_e2e += 1.

#### Уровень 3 — Full scope E2E

**Условие:** срабатывает ЛЮБОЙ из триггеров:
1. Счётчик merges_without_full_e2e ≥ 5
2. Завершение Stage из мастер-плана (любого)
3. Изменения в auth/, middleware, RLS-политиках, api/ routes
4. Изменения затрагивают ≥3 модуля одновременно
5. Пользователь явно запросил full scope

**Действие:** `npx playwright test` (все файлы). Счётчик merges_without_full_e2e = 0.

### Счётчик merges_without_full_e2e

Хранится в `PROJECT_CONTEXT.md` в секции «Текущий статус» как строка:
```
**E2E full scope:** последний — [дата], merge без full scope с тех пор: [N]
```

Claude Code обновляет эту строку при каждом merge через /update-docs:
- Уровень 1 или 2 — инкрементировать N
- Уровень 3 — записать текущую дату, сбросить N = 0

### Нарастание тестов по этапам

Тесты накапливаются: Stage N пишет свои + прогоняет все предыдущие при full scope.
Если Stage N сломал сценарий из Stage N-1 — full scope E2E поймает, merge заблокирован.
