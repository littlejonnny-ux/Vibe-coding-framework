# Post-Merge Retrospective

Этот механизм запускается автоматически после каждого merge. Claude Code анализирует, какие skills и agents были активированы, оценивает их необходимость, и фиксирует маркеры для оптимизации будущих решений.

---

## Алгоритм

### 1. Сбор данных
```bash
git diff origin/main~1..origin/main --stat    # размер PR
git diff origin/main~1..origin/main --name-only  # затронутые файлы
```
- Подсчёт: insertions + deletions в .ts/.tsx/.js/.jsx
- Тип изменений: code / docs / css / mixed

### 2. Анализ активаций
- Какие skills были активированы в этой сессии
- Какие agents были вызваны
- Был ли /code-review, /update-docs, tdd, security-review
- Были ли написаны E2E-тесты для новых UI-механик
- Какой уровень E2E был применён (1/2/3) и почему
- Проходят ли все существующие E2E-тесты (регрессия)

### 3. Оценка необходимости
Для каждого активированного механизма:
- **Был ли результат?** (Нашёл ли code-reviewer blocking issues? Нашёл ли security-reviewer уязвимости?)
- **Был ли результат ожидаем?** (Для PR с auth-кодом security-review ожидаемо полезен)
- **Было ли это пропорционально?** (Мультиагентный review на 10-строчный фикс = непропорционально)

### 4. Фиксация
Если обнаружена избыточная активация — записать маркер в LEARNED_OVERRIDES.md.

### 5. Извлечение технических паттернов
После фиксации маркеров — проанализировать сессию на нетривиальные технические открытия:
- Нестандартное поведение API или библиотеки (не то, что написано в docs)
- Workaround для известного бага или ограничения
- Нетривиальная TypeScript-конструкция, решившая задачу
- Паттерн, который окажется полезным в другом проекте

**Если есть хотя бы один паттерн** — записать в `LEARNED_PATTERNS.md`:
```markdown
### [ДАТА] [Название паттерна]
- **Область:** [библиотека / стек / паттерн]
- **Паттерн:** [что именно]
- **Почему нетривиально:** [в чём неочевидность]
- **Пример:** [краткий код или шаги]
```

**Если паттернов нет** — не записывать ничего (не добавлять пустые записи).

---

## Формат записи в LEARNED_OVERRIDES.md

```markdown
### [ДАТА] PR #N — [краткое описание]
- **Diff:** +X / -Y строк, Z файлов
- **Тип:** code / docs / css / mixed
- **Активировано:** [список skills и agents]
- **Результативно:** [что нашло реальные проблемы]
- **Избыточно:** [что не нашло ничего / было непропорционально]
- **Маркер:** [правило для будущего]
```

---

## Пример

```markdown
### 2026-03-27 PR #32 — Добавить фильтр по дате на странице архива
- **Diff:** +28 / -3 строк, 1 файл (ArchivePage.tsx)
- **Тип:** code (UI)
- **Активировано:** code-reviewer (3 агента), security-reviewer
- **Результативно:** code-reviewer — 0 issues. security-reviewer — 0 issues.
- **Избыточно:** Оба review-агента. Задача — добавление select-фильтра в существующую таблицу.
- **Маркер:** UI-фикс в 1 файле <50 строк, без auth/input → пропускать /code-review и security-reviewer.
```

---

## Когда НЕ проводить ретроспективу

- Docs-only изменения
- Пустые merge (только .gitkeep, конфиги)
- Тир LITE (ретроспектива добавлена только в STANDARD и ENTERPRISE)

---

## POST-MERGE BACKPORT

После завершения ретроспективы (только STANDARD и ENTERPRISE):

1. Запустить `node .claude/hooks/scripts/backport-analyzer.js`
2. Показать результат анализа пользователю
3. Дождаться решения: «перенеси в framework» или «пропустить»
4. Если подтверждено — выполнить backport в Vibe-Coding-Framework репо

**Скрипт автоматически пропускает LITE тир.** Для STANDARD/ENTERPRISE запускается всегда.
