# Карта триггеров

Этот файл определяет, какие skills, agents и commands активировать для какого типа задачи. Claude Code сопоставляет описание задачи с таблицами ниже и активирует минимально необходимый набор.

**ВАЖНО:** Перед активацией проверь LEARNED_OVERRIDES.md — маркеры из него ПЕРЕОПРЕДЕЛЯЮТ правила из этого файла.

---

## Триггеры активации contexts (режимы работы)

Contexts — это файлы поведения Claude Code в разных режимах. Активируются в начале сессии или при явном переключении режима.

| Сигнал | Context | Когда |
|--------|---------|-------|
| Пользователь описывает новую фичу / задачу | `contexts/dev.md` | По умолчанию для любой задачи разработки |
| Пользователь просит "проверь", "проревьюй", "посмотри на код" | `contexts/review.md` | При явном запросе ревью |
| Пользователь просит "исследуй", "разберись", "найди", "изучи" | `contexts/research.md` | При исследовательских задачах |

**Правило:** `contexts/dev.md` — дефолтный режим для ВСЕХ задач разработки. Остальные два — по явному сигналу.

## Триггеры активации agents

| Ситуация | Agent | Тир |
|---|---|---|
| Задача затрагивает >3 файла, неочевидная архитектура | planner | STD+ |
| PR содержит код (не docs/CSS only) + соответствует правилам /code-review | code-reviewer | STD+ |
| PR содержит auth/RLS/API/формы с пользовательским вводом | security-reviewer | STD+ |
| PR содержит Supabase queries/RLS/миграции | database-reviewer | STD+ |
| `npm run build` failed | build-error-resolver | ALL |

## Триггеры активации commands

| Ситуация | Command | Тир |
|---|---|---|
| Сложная задача, нужна декомпозиция | /plan | STD+ |
| После push, перед merge (если обязателен) | /code-review | STD+ |
| После merge | /update-docs | STD+ |
| Build failed | /build-fix | ALL |
| Бизнес-логика, формулы, серверные routes | /tdd | STD+ |
| Перед merge крупного PR (>200 строк) | /verify | ENT |
| Сессия длинная, context забит | /compact | STD+ |

## Правила «когда НЕ активировать» (сохранение качества контекста и usage limits)

| Ситуация | Пропускаемые механизмы |
|---|---|
| Docs-only изменения (.md файлы) | /code-review, security-reviewer, tdd, tests |
| CSS-only правки | /code-review, security-reviewer, tdd, tests |
| Однофайловый фикс <50 строк с очевидным scope | planner, verification-loop |
| Текстовые правки (labels, placeholder, tooltip) | Все кроме lint + commit |
| Правка в одном компоненте <50 строк | planner |
| Добавление/изменение .gitignore, README, конфигов | Все кроме lint + commit |
| Изменения только в calculations.ts, constants.ts, utils.ts (без UI) | E2E-тесты (покрыто unit-тестами) |
| Изменения только в data layer (hooks/) без нового UI | E2E-тесты |

## Триггеры Ultraplan (предлагать до начала реализации)

Ultraplan — облачное планирование с браузерным ревью плана. Предлагается когда задача
соответствует хотя бы **2 критериям из списка A** или **1 критерию из списка B**.

### Список A — количественные сигналы (считаем сколько совпадает)

| Сигнал в описании задачи | Вес |
|---|---|
| Затрагивает или упоминает >5 файлов / компонентов | +1 |
| Новая сущность БД (таблица, новые колонки в 2+ таблицах) | +1 |
| Новый пользовательский flow (многошаговый процесс, wizard, согласование) | +1 |
| Изменение архитектуры (новый слой, новый сервис, переход на другой паттерн) | +1 |
| Задача описана в >5 предложениях с несколькими «и», «также», «плюс» | +1 |
| Упоминание интеграции с внешним сервисом (API, webhook, OAuth) | +1 |
| Затрагивает auth, роли, права доступа | +1 |
| Упоминание нескольких страниц/экранов одновременно | +1 |

**Порог:** ≥2 совпадений — предложить Ultraplan

### Список B — качественные сигналы (достаточно одного)

| Сигнал | Описание |
|---|---|
| «С нуля», «новый модуль», «новый раздел» | Создание чего-то принципиально нового |
| «Переработать», «переписать», «мигрировать» | Рефакторинг существующей системы |
| «Система», «движок», «механизм» | Задача системного уровня |
| «Этап», «фаза», «milestone» | Пользователь сам идентифицирует крупность |
| Задача описывает бизнес-процесс целиком (от начала до конца) | Сквозной flow |

**Порог:** ≥1 совпадения — предложить Ultraplan

### Как предлагать Ultraplan

Когда триггер сработал — **до начала любой реализации** скажи пользователю:

```
⚡ Ultraplan рекомендован

Эта задача соответствует критериям сложности: [перечисли какие именно сработали].

Ultraplan составит детальный план в облаке пока ты работаешь в терминале,
и даст возможность просмотреть его в браузере до начала реализации.

Запустить? — напиши `/ultraplan [твоё описание задачи]`
Продолжить без Ultraplan? — напиши «продолжай»
```

### Когда НЕ предлагать Ultraplan

| Ситуация | Причина |
|---|---|
| Пользователь уже написал `/ultraplan` | Уже запущен |
| Задача соответствует правилам «когда НЕ активировать» ниже | Слишком мелкая |
| Пользователь явно написал «быстро», «мелочь», «просто» | Выражена воля |
| Тир проекта LITE | Ultraplan избыточен для простых проектов |
| Предыдущая задача в сессии уже прошла через Ultraplan | Не повторять в той же сессии |

## Принцип минимальной активации

Claude Code всегда стремится к минимальному набору механизмов, достаточному для качественного выполнения задачи. Если задача простая — не нужно активировать сложные механизмы только потому что они доступны.

**Базовая формула:** lint + commit + push — это минимум для любой задачи в любом тире. Всё остальное — по триггерам.
